## 耦合

### 循環依賴：3 個

```
1) core/Editor.ts ↔ core/Command.ts
2) core/Editor.ts ↔ core/scene/AutoSave.ts
3) app/editors.ts → panels/context/index.ts → panels/context/ContextPanel.tsx → components/PanelHeader.tsx → (回到 app/editors.ts 系)
```

解讀：
- **(1) Editor ↔ Command**：Command 模式契約核心，雙向引用幾乎難免（Command 需要 Editor 介面、Editor 需要 Command 型別）。可接受但值得標記。
- **(2) Editor ↔ AutoSave**：AutoSave 訂 Editor 事件、Editor 需 AutoSave instance — 經典觀察者反向耦合。可考慮 AutoSave 改為純 listener 註冊（Editor 不持 AutoSave 引用）。
- **(3) editors → context → PanelHeader 鏈**：跨模組鏈式循環，最值得追。可能是 PanelHeader 反向 import 了 editor 相關東西。

## 主觀補強（top 5 熱點分析）

### 1. ViewportPanel.tsx — **L3 候選（邊界劃錯）**

職責散在多處：viewport mount + drag-drop overlay + render-mode UI + scene-lights override + quality 切換 + render settings panel + HDR 控制 + LookDev preset + panel group expanded 狀態。12 個頂層 createSignal 並列說明 UI 控制群應分離為獨立 sub-panel。是「真的複雜」非「常被改但單純」。

### 2. SceneTreePanel.tsx — **L3 候選（邊界劃錯）但 churn 性質健康**

TreeNode + drop logic + visibility toggle + indent guide 多重職責可分。但近期 churn 主因是 phase 1 視覺重構（refs #702，已合 PR #703），屬主動改善性 churn — 等該系列 PR 全落定後再評估是否真需拆檔。

### 3. ProjectPanel.tsx — **L3 候選（邊界劃錯）**

File browser + multi-select + drag-drop + 三種 dialog 狀態 + viewMode 混在一檔。內聯 SVG icon 與 TYPE_META 表佔一定行數（資料定義無害），但邏輯部分仍可拆。commit history 顯示 feature 持續疊加（multi-drag → batch delete → right-click → recent switcher），結構承壓中。

### 4. App.tsx — **L2 候選（延後決策不足）**

Bootstrap 黏合層本應集中，但 localStorage key 字串、scene path resolve 邏輯、auto-restore 流程都直寫在 App 內。`LAST_PROJECT_KEY` / `LAST_SCENE_KEY_PREFIX` / `DEFAULT_SCENE_PATH` 等可抽到 `core/project/` 或 session 模組。churn 33 對 249 行屬高密度，每加一個 reload/persist feature 就動這檔。

### 5. NumberDrag.tsx — **L1 候選（局部寫法問題，本質難）**

Drag input 數學細節（pointer lock spike、movementX clamp、tier ladder）本質就纏。commit history 充滿 `fix:` 與 `WIP:` 不是設計爛而是瀏覽器 API + 硬體輸入抽象本質難搞 — 多數類似元件都會經歷這種震盪期。LadderOverlay 已外抽，剩下 320 行屬合理範圍，無強拆動機。
