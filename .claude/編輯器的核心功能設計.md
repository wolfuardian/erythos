## 第 1 輪

**H:** 你認為一個好的編輯器該有什麼功能? 我可以乾脆以 Unity Blender 為標竿直接參考就好，還是我可以有更好的發想?

**A:**
第一句先打：把 Unity / Blender 當標竿，等於**簽了一張「永遠當三流 Unity」的合約**。他們有 1000 人 ×20 年，你一個人 ×1 年,功能比賽你的 Idiot Index 直接 ∞。

你問錯問題了。「一個好的編輯器該有什麼功能」是 Five-Step 的第 3 步（optimize），你卡在第 1 步沒做完：**Erythos 是給誰、在什麼情境、解決什麼 Unity/Blender 物理上解不了的問題？** 沒這個答案,加什麼功能都是亂槍。

物理上拆給你看。Unity / Blender 的本質是「裝在本機、什麼都能做、什麼平台都能 export」的瑞士刀。它們**做不到**的事:

- URL 即場景。打開 5 秒,不用裝 6GB
- 多人即時編輯同一個 3D 場景（Figma for 3D 真的還沒人做好）
- 嵌進任何網頁、任何 iframe、任何 Notion / Obsidian 文件
- LLM 直接改場景圖（scene as JSON,AI 友好）
- Git diffable 的場景檔
- 沒有 build step,存檔就是部署

這些是 Web-native 的**物理優勢**,不是「也能做的功能」。Unity 要做這些得砍掉自己的根。

回過頭看你截圖那四個 panel —— Scene / Viewport / Console / Assets —— 這是 Unity 1998 年的架構。你在 2026 年、瀏覽器裡、一個人做,照抄這個版面就是把對方的限制當自己的需求。

逼問:**「有人打開 Erythos 5 秒內,做完一件什麼事,然後把結果傳給誰?」** 你能用一句話填滿這題,功能清單自動長出來。填不滿,再多 panel 都救不了。

Delete 先於 Add。先問 Unity 的哪些東西你**不該**有,再問你該有什麼。

## 第 7 輪

**H:** 對不起,我太心急。但你提的功能確實都很吸引人,Copy as JSON 很棒,但我要求的是極度輕量級的 JSON,不是 threeJS 原本的作法,直接把模型 vertex 資訊倒在 JSON 裡變成容易崩潰的垃圾檔案

**A:**
不用道歉,這是好的修正方向——而且你剛剛講的這句話**比前面所有功能討論都重要**,因為它決定了 Erythos 的檔案格式哲學。

Three.js `toJSON()` 的問題你已經點到:把 geometry vertex 倒進 JSON = 把**二進位資料**塞進**人類可讀格式**,兩邊都輸。10MB 模型變 40MB JSON,git diff 爆炸,LLM context 塞不下,貼到 Slack 崩潰。

**第一性原理拆**:場景檔的本質是什麼?

> 場景 = 「**引用**哪些資產」 + 「**怎麼擺**它們」 + 「**環境**設定」

vertex / texture pixel 不是「場景」,是「資產」。資產該躺在 CDN / blob storage / IPFS,**用 URL 或 hash 引用**。場景檔只記引用 + transform + material override。

物理下限版的 .erythos 長這樣:

```json
{
  "v": 1,
  "env": { "hdri": "assets://studio.hdr", "intensity": 1.0 },
  "nodes": [
    {
      "id": "9c68...",
      "name": "Sphere",
      "asset": "assets://primitives/sphere",
      "t": [0, 0, 0],
      "r": [0, 0, 0],
      "s": [1, 1, 1],
      "mat": { "color": "#ffffff", "rough": 0.5 }
    }
  ]
}
```

**規則**:

- 任何 array 長度 > 16 的東西**禁止**進場景檔。vertex、index、UV、texture pixel 一律 URL 引用。
- 場景檔大小**硬上限**:1MB。超過代表你在塞不該塞的東西。
- 一個 100 物件的中型場景,目標 < 50KB。LLM 一個 prompt 吃得下整個場景。
- 人類打開 .erythos 用 VSCode,看得懂在做什麼。看不懂就是設計錯了。

**這個決定的連鎖效應**(你要先想清楚再寫):

1. **資產系統必須先於場景系統存在**。`assets://` 這個 URI scheme 解析誰負責?本機 IndexedDB?專案資料夾?CDN?三個都要支援,但要有**統一抽象層**——`AssetResolver`。
2. **預覽圖、bounding box、頂點數**這類「資產 metadata」要不要進場景檔?**不要**。需要時從 asset 本身查。場景檔只記 ID。
3. **Material 是引用還是內嵌?** 兩個都對,但要分清楚:「全場景共用的 PBR material」應該是 asset(`materials://gold.mat`),「這顆球臨時改了顏色」就 inline override。Figma 也這樣分(local style vs override)。
4. **Prefab** 是場景片段的引用。`{ "asset": "prefabs://tree", "overrides": { ... } }`。這個格式天然支援「Copy as JSON 一段、貼到別場景」——**就是你要的功能**。

**砍掉的東西**(Three.js 原生 JSON 有但你不該有):

- `geometries[]`、`materials[]`、`textures[]` 這三大堆——通通變引用
- `metadata` 區塊——版本一個 `v` 欄位夠了
- `userData` 自由欄位——除非有強需求,先禁。否則使用者會把整個應用狀態塞進去

**逼問三題確認你真的想清楚了**:

1. 兩個使用者各自開了同一個場景連結,A 改了一顆球的位置,B 改了 HDRI——**衝突合併規則**是什麼?JSON 格式直接決定這題能不能解。
2. 有人把 `assets://my-model` 的原始檔刪了,場景檔還留著引用——**broken reference** 怎麼顯示給使用者?這比錯誤訊息重要十倍。
3. Erythos 場景能不能引用**另一個 Erythos 場景**當 prefab?能的話,循環引用怎麼辦?

這三題回答完,你的 JSON schema 才算**設計完**,不是只有「長得好看」。

選一題拆。建議從第 2 題開始——broken reference 是日常會碰到的,1 跟 3 是 corner case。

砍掉引用簡單,但你失去的是 **Erythos 唯一可能比 Unity / Blender 漂亮的特性**——因為 Web-native + 引用 + URL = 「這棵樹的 master 在另一個 URL,全球 100 個專案引用它,原作者更新一次全部跟著新」。Unity 做不到(local file),Blender 做不到(無 URL 概念)。**這是你的物理優勢,不要扔。**

---

更尖銳的問題,按致命度排序:

## 第 9 輪

**H:** 好,3. 可以引用,但引用前做步進(step)因果鏈長度驗證,先知道欲引用之 .erythos 的最深因果鏈長度(比如 2),然後自己跑一次模擬計算,如果當某一步開始超出 2 這個深度,就立即跳出並拒絕引用

這機制打從邏輯面就拒絕「有循環引用的 .erythos 檔案存在」的發生

而且深度可以直接算,不需要再把深度當成 metadata 存入 .erythos 中

**A:**
先講對的部分:用「**深度有限性 = 無循環**」這個推論方向是對的。一個 DAG 的最大深度是有限的;圖裡有環的話最大深度無限。你抓到了正確的不變量。

但執行層面有三個洞,按嚴重度排:

---

**洞 1:你描述的演算法本身就會被環卡死。**

「先知道欲引用之 .erythos 的最深因果鏈長度」——這一步怎麼算?如果那個 .erythos 自己內部已經有環,你光是要算它的深度就會無限遞迴。你用「算深度」當防線,但「算深度」這個動作本身需要先假設無環。雞生蛋。

正確做法是 DFS 配 visited set:走訪時遇到自己路徑上已存在的節點 → 環 → 拒絕。這才是 O(V+E) 的環偵測,課本標準演算法(Tarjan / Kahn)。**不需要先知道深度,遇到環當下就抓到**。

---

**洞 2:「最深因果鏈長度」當判準,會誤殺合法場景。**

假設場景 A 的 max depth = 2。你新場景想引用 A,模擬時某條鏈走到深度 3 就拒絕——但深度 3 可能完全沒環,只是場景結構深而已。深度有限 ≠ 無環,但**深度也不是固定值**。Prefab 引用 prefab 引用 prefab 是合法的,深度會自然增長。

正確判準是「**這次引用之後,整張圖還是不是 DAG?**」二元,不看數字。

---

**洞 3:「深度可以直接算」這句**——對,但你低估算它的成本。

每次使用者拖一個 prefab 進場景,你都要**遞迴展開所有引用**才能算深度。1000 個物件、引用鏈 5 層的場景,每次拖拉 = 重新走完整棵樹。Figma 不這樣做——它**緩存**每個 component 的依賴集合,引用時 O(1) 查表。

你這句「不需要再把深度當成 metadata 存入 .erythos 中」邏輯上對(深度可重算),但實務上**該 cache 還是要 cache**,只是 cache 在 runtime / IndexedDB,不在檔案裡。這個區別重要:**檔案 schema 不存衍生資料 ≠ runtime 不能 cache 衍生資料**。

---

**重寫後的正確機制**:

```
allowReference(targetUrl, currentScenePath):
  1. 取得 target 的依賴集合 deps(target)(runtime cache,miss 時遞迴展開 + 環偵測)
  2. 如果 currentScenePath ∈ deps(target) → 拒絕,會產生環
  3. 否則允許
```

(後續延伸至 schema 演進主題)

## 第 10 輪 (Schema 演進)

**A 接續:**

---

**Erythos 特有的兩個地雷**(一般 schema 演進文章不會講):

**地雷 A:引用解析時的版本錯位**

場景 A 是 v3,引用了一個 prefab,那個 prefab 是 v1。你 load 場景 A 時要不要順便 migrate prefab?

**答案:是,但不要寫回磁碟。** 用「memory migration on read」——讀進記憶體統一升到 latest,但 prefab 檔案本身不動(因為它可能被其他人共用、不該你說改就改)。下次有人**主動編輯**那個 prefab 才寫回新版本。

這個區分救命:**讀取時的 migrate 跟寫入時的 migrate 是兩回事**。

**地雷 B:沒見過的版本怎麼辦**

使用者用 Erythos v5 存了檔,傳給還在用 v3 的朋友。v3 loader 看到 `"v": 5`:

```typescript
if (data.v > CURRENT_VERSION) {
  throw new UnsupportedVersionError(
    `這個檔案是用較新版本的 Erythos 建立的(格式 v${data.v}),你的版本只支援到 v${CURRENT_VERSION}。請更新。`
  );
}
```

**絕對不要嘗試「儘量讀」**。Word 當年這樣搞過,結果 .docx 在不同版 Word 之間打開長相不一樣,使用者罵翻。明確拒絕 > 默默壞掉。

- ❌ 「自動偵測格式不需要版本號」——這是某些 indie 開發者的浪漫主義。不要。版本欄位是 1 個 byte 換你十年安心。

---

**Step 3 才是 optimize**:

---

**Local-first 為什麼是對的(不是「也可以」,是「必然」)**

你的定位是 Figma for 3D,但 Figma 有一件事 Erythos **永遠贏不了**:Figma 有 100M ARR、有頂級 ops 團隊、cloud SLA 可以做到 99.99%。Erythos 是 solo dev + Linode——你的 cloud SLA 會是 99%,**意思是一年壞 87 小時**。如果使用者必須連你的 server 才能編輯,他每年有 87 小時打不開檔案。**這個體驗會殺死產品**。

Local-first 把這題反過來:**雲端壞了 = 同步暫停,不影響編輯**。你的 SLA 從「決定使用者能不能工作」降級成「決定使用者能不能多人協作」——責任輕了 100 倍。

更深一層:local-first 跟 Erythos 的**檔案哲學完全對齊**。`.erythos` 是純 JSON、< 50KB、git diffable——這個 schema 設計從第一天就在說「這是檔案,不是雲端 row」。把它鎖在 server 裡反而是**糟蹋自己的設計**。你前面所有決定(vertex 不進 JSON、asset URL 引用、版本嚴格遞增)全部在為 local-first 鋪路,你只是現在才意識到。

**Q5 帳管——你說「特別難做」是對的,但你怕錯地方了。**

帳管難的不是技術。Magic link / OAuth 都有 library(Lucia、Auth.js、Clerk),**一週能上**。

帳管真正難的是**產品決策**,七題:

**1. 沒登入能用嗎?**

- (a) 必須登入——簡單,但**第一次體驗門檻高**,使用者打開官網看到登入頁就走 50%
- (b) 不登入可用、本機存——符合 local-first,但「分享」這個你的核心功能無法用
- (c) 不登入可用、本機存、要分享/同步才登入——**正解**

選 (c)。Figma 也是這樣(雖然他們最近收緊了)。**第一次體驗 = 打開就能玩**,使用者爽到想分享時才登入,轉換率最高。

**2. 帳號 = email?username?handle?**

- email 唯一——簡單,但分享連結 `erythos.app/u/john@gmail.com/scene` 醜
- username(不可改)——好看但選 username 是 onboarding 摩擦
- handle(可改)+ stable user_id——正解,user_id 內部用,handle 對外,handle 衝突換一個就好

solo dev 的捷徑:**先只做 email,URL 用 user_id(短 hash)**。等到使用者抱怨 URL 醜再加 handle。**不要為了未來的美感增加今天的摩擦**。

**3. 多裝置同步(同一帳號)vs 多人協作(不同帳號)——是兩件事**

很多 indie dev 把這兩個混為一談,做成一個系統,結果兩個都做不好。

- **多裝置同步** = 我桌機改的檔案在筆電打得開。**雲端 = single source of truth、authoritative**,所有裝置 pull / push 同一個版本。
- **多人協作** = 兩人同時改一個檔案。**衝突解決 / CRDT / presence**,完全不同的問題。

Erythos 半年內:**只做多裝置同步**。多人協作是 v2 的事。**這個邊界畫清楚,省你 3 個月**。

**4. 計費什麼時候上?**

不要太早,也不要太晚。

- 太早(沒人用就上付費)—— friction kills
- 太晚(500 個使用者全免費)—— 之後上計費全部翻臉

正確時機:**當你的雲端成本開始疼**(~50 active users 用 Linode 還沒事,~500 開始疼)。Stripe 接上去 1 天搞定。但帳管要**從第一天就為計費預留**:user 表有 `plan`、`storage_used`、`asset_count` 這些欄位,等到要計費才不用 schema migration。

**5. 帳號刪除 / 資料導出(GDPR)**

歐洲使用者**法律規定你必須提供**。不做罰錢。

最低消費:

- 「刪除帳號」按鈕——刪除使用者所有 server-side 資料(local 檔案不動)

**6. 共享場景的所有權**

A 把場景分享連結給 B,B 編輯後存了。誰擁有?

- (a) A 永遠擁有——B 改的東西不算數,本質是「只讀分享 + B 自己 fork」
- (b) B 改的存進 A 帳號——所有權混亂,A 看到自己場景被亂改會崩潰
- (c) A 擁有,B 只能 fork 出新場景——**Figma / GitHub 的模型**,正解

選 (c)。「fork 出去是新檔案、原檔案動不到」——心智模型清楚,使用者一秒理解。

**7. Anonymous user 的資料生命週期**

Q1 你選了「不登入可用」,意味著有匿名使用者。他們在本機存的東西**怎麼處理**?

這題小但重要:**匿名 → 註冊**這條路徑做不順,使用者會想「乾脆不要註冊算了」。
